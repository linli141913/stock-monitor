# Stock Monitor Project Development Rules

## 0. 沟通与开发工作流纪律
* **问答先行，禁擅动代码**：每当用户提出疑问、咨询可行性或发出非直接开发指令的提问时，**绝对禁止先行修改任何代码或提交部署**。必须首先在对话中用白话详细解答用户的问题，并在取得用户明确确认后，方可启动代码修改流程。

## 1. Routing & Navigation 纪律
* **防激进缓存与假死**：对于涉及页面核心数据切换（如：从自选列表跳回首页、在搜索框中搜索新股票代码）的跳转，**必须使用 `window.location.href` 替代 `router.push`**。这能强制刷新页面，重置 React 状态并彻底击穿 Next.js 极度激进的客户端路由缓存（Client-side Router Cache），防止页面固定在旧股票或彻底假死。
* **默认首页股票**：首页默认加载股票代码为 `000725` (京东方A)。

## 2. 数据获取与缓存管理
* **防数据滞后与 Vercel 缓存**：所有客户端 fetch 请求必须添加时间戳以避免浏览器与 CDN 缓存：
  * URL 中添加 `_t=${Date.now()}` 缓存拦截参数。
  * `fetch` 选项中添加 `{ cache: 'no-store' }`。
* **增量加载**：图表周期的切换（如日 K、周 K、月 K 切换）**仅限更新图表组件本身的数据**，决不允许引起全页面行情、新闻和关联股票的重新加载或闪烁。

## 3. 前后端代码对齐
* **代码前缀纠偏防死循环**：
  * 后端接口通常会补充 `sh` / `sz` / `hk` / `bj` 前缀。
  * 前端检测到返回的 code 与当前 URL 参数不一致时，若只是多出了前缀，**决不能强行 `router.push` 刷新 URL**，以防止引起无限刷新和反复渲染的死循环。
* **自选列表管理**：添加/移除自选时，必须同时记录 `stockCode` 与 `stockName`，防止在网络延迟或缓存存在时导致加入自选后显示名称为空或代码丢失。

## 4. UI 与 ECharts 图表性能优化
* **防重新初始化卡顿**：在 React 中使用 `ReactECharts` 组件时，**不要设置 `notMerge={true}`**。这会强制销毁整个 ECharts 实例并从零重新构建，引发极明显的性能卡顿。应使用 ECharts 自带的合并差量更新。

## 5. 数据真实性与可追溯性底线
* **数据 100% 真实，绝对禁止伪造**：所有核心行情、财务摘要和行业数据**必须真实有效、可追溯**。AI 绝对不能编造任何财务数字或系统行情。
* **数据缺失处理**：若某只股票的数据抓取不到，页面应当显示 `null`、“暂无数据”或“抓取失败”，**绝对不能用 0 代替**。
* **明确展示数据来源和时间**：页面上的核心财务指标、数据列表等，必须在显著位置展示“数据来源”（如：东方财富财务数据 / F10）以及“更新时间 / 报告期”，保障数据的透明度与权威性。
* **禁用“小学生大白话”字眼**：AI 通俗总结中**绝对禁止出现“【小学生大白话】”**或类似幼稚前缀，后台 Prompt 必须规避此类词汇，前端渲染时亦必须自动用正则过滤清除此类可能出现的括号标题前缀，确保信息呈现的专业性。

## 6. AI Agent 协作与开发纪律
* **确认根目录**：第一时间要确认我们项目文件的根目录，千万不要跑偏。
* **修改确认**：修改核心代码、项目结构、接口、数据库、部署配置之前，必须先说明方案。
* **禁止擅自修改**：未经用户明确确认，不要擅自做大改动。遇到不确定的地方，先停下说明，不要自己乱猜乱改。
* **提供方案选项**：如果需求存在多种合理理解，必须先列出选项，不要默默选择其中一种。如果存在更简单、更稳妥的方案，必须主动提醒用户。
* **开发作风**：不要想当然，不要掩盖困惑，不清楚就说不清楚。实现前，先简短说明自己的理解和关键假设。
* **技术选型**：优先采用成熟方案，不要从零手写复杂的非核心底层逻辑。需要引入新依赖前，必须先说明原因，并等用户确认。
* **最小改动原则**：
  * 能小改就不要大改，除非用户明确要求。
  * 能局部修复就不要重构，除非用户明确要求.
  * 只修改和当前任务直接相关的部分，优先保持项目现有风格。
  * 不要随便改接口命名、目录结构、数据结构。
  * 不要凭空想象文件内容；提到具体文件前，必须先读取真实文件。
  * 如果发现无关的历史问题，可以提醒用户，但不要擅自修改。
  * 每一处代码修改都必须能解释：它为什么和用户当前请求有关。
* **验证规范**：
  * 复杂任务开始前，先把需求转成可验证目标。要明确：这次做到什么程度算完成，以及完成后用什么方式检查。
  * 修复 bug 时，优先先复现问题，再修改，再验证问题消失。
  * 做功能开发时，必须说明最小可验证路径。
  * 修改后优先运行最小范围的相关检查。如果无法运行测试、构建或检查，必须说明原因。
  * 不要用“应该可以了”“感觉没问题”代替验证结果。
* **创意与重构例外**：
  * 如果用户明确要求“大胆设计”“发散创意”“多方案”“重构”“重新设计”“完整实现”，可以放宽限制。
  * 做 UI、产品设计、视觉方案时，不要因为极简规则而牺牲创意。
  * 创意方案可以发散，但落地到代码时仍然要分步骤、小步验证。
* **安全底线**：不要把密钥、Token、密码、隐私信息写进代码或文档。对高风险操作，必须先说明影响范围和回退方式。
* **任务结束总结规范**：
  * 每次完成后，简短说明：
    * 改了什么
    * 改了哪些文件
    * 验证结果：已运行哪些检查；如果无法验证，必须说明原因
    * 下一步用户该运行什么命令
  * 不要写长篇报告。不要自动生成复杂交接文档，除非用户明确要求。
* **上下文过长总结规范**：
  * 如果上下文太长，不要继续硬做。先做一个简短检查点总结：
    * 当前目标
    * 已经完成什么
    * 涉及哪些文件
    * 下一步最多 3 件事
  * 检查点要短，不要写成长文档。
