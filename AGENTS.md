## 1. 当前产品定位、生产基线与升级边界

本项目当前是“准实时股票监测 + 强提醒 + 行业洞察 + AI 证据解释”系统。

V5.0 计划在现有生产系统基础上新增市场环境、主线行业、行业 ETF 观察池、三级龙头规则状态、状态机和雷达历史验证。以上属于待实施范围，不得写成当前已经上线的能力。

本项目不是自动交易系统，不承诺毫秒级实时行情，不提供自动买卖服务。

当前生产技术基线：

- 前端：Next.js 16、React 19、TypeScript、ECharts、CSS Modules。
- 后端：Python、FastAPI、Pydantic、APScheduler。
- 数据：SQLite；腾讯财经、东方财富、AKShare、新浪财经、巨潮资讯等公开数据源。
- AI：OpenAI 兼容 API，通过环境变量配置模型、地址和密钥。
- 部署：Vercel 承载前端；后端当前通过独立服务地址连接，不属于 Vercel 前端部署的一部分。

以上是当前生产基线，不代表 V5.0 规划中的新组件被禁止。Polars、Parquet、DuckDB、PostgreSQL、Redis、Dramatiq、pgvector 等新技术必须先证明必要性，说明用途、收益、影响和替代方案，经用户确认后分阶段引入；现有 SQLite、pandas 或已有模块能满足需求时，优先保留或增强，不强制重写。

固定本地端口：

- 前端本地入口固定为 `http://localhost:4000`，`stock-monitor` 的 `dev` 和 `start` 命令必须使用端口 `4000`。
- 后端本地入口固定为 `http://127.0.0.1:8001`，前端服务端代理和后端自调用必须使用端口 `8001`。
- 未经用户在当前对话明确确认，不得改用 `4001`、`8000`、`8002` 或其他端口。

## 2. 操作权限分层

- 当用户要求“查看、检查、分析、总结、排查”时，可以直接执行与问题相关的只读操作，例如读取代码、查看 Git 状态、检查日志结构、运行不会写入业务数据的静态检查。

- V5.0 第一次进入实施阶段时，修改代码前必须先报告：当前真实运行状态、PRD 与代码差异、升级规划与实现差异、保留/增强/新增/迁移/替换/暂缓清单、字段级数据源合同、容量与性能估算、测试/备份/回滚方案。

### 2.2 本地修改

- 修改核心代码、接口、数据库结构、项目目录、依赖、部署配置前，必须先说明方案和验证方式，并获得用户清晰、无歧义的执行授权。
- “同意”“执行”“修改”“开始吧”“按方案做”“继续处理”等明确表达均可视为授权，不机械限定固定词语。

- 坚持最小修改：只改当前目标直接相关的代码，不顺手优化、重构或格式化无关文件。

- 本地修改不等于允许 Git commit、推送 GitHub、触发 Vercel、修改线上环境变量、重启线上后端或迁移生产数据库。

## 3. 项目目录与渐进式架构边界

当前主要目录：

- `stock-monitor/`：Next.js 前端。
- `backend/`：FastAPI 后端、数据抓取、AI 分析、定时任务和 SQLite。
- `AGENTS.md`：本项目唯一长期协作规则。

后续整理遵循以下边界，但不得为了目录好看而一次性大重构：

- 前端页面只负责编排；通用请求、格式化和类型定义逐步集中到 `lib/`、`types/` 或对应业务目录。
- 后端路由、外部数据源、业务逻辑、数据库访问、定时任务逐步分离；现有接口路径在没有明确批准时保持兼容。

- 不新增依赖，除非现有技术栈无法安全完成任务，并且已经说明原因、影响和替代方案。
- 已经上线并稳定工作的功能、接口兼容性和生产数据必须保留；“保留”不等于代码完全不可修改，经当前阶段授权后可以做必要的兼容性增强。
- V5.0 新功能必须作为独立模块增量接入现有项目，不得另建重复项目。
- 新功能修改现有模块前，必须标明属于“保留、增强、新增、迁移、替换、暂缓”中的哪一类。

## 4. Routing 与前端状态

- 首页默认股票代码为 `000725`（京东方A）。
- 股票代码变化且存在旧 React 状态或 Next.js 路由缓存污染风险时，使用 `window.location.href` 强制刷新。
- 普通页面导航使用 Next.js `Link`；不要把所有导航一律改成整页刷新。
- 后端返回的代码仅多出 `sh`、`sz`、`hk`、`bj` 前缀时，不得重复刷新 URL，避免死循环。
- 切换 K 线周期只更新图表数据，不重新加载全页行情、新闻、行业和相关股票。
- `ReactECharts` 默认使用差量合并更新，不设置 `notMerge={true}`；确有例外必须说明原因并验证性能。

## 5. 数据获取、刷新与缓存

- 行情等高频接口：URL 添加 `_t=${Date.now()}`，并使用 `{ cache: 'no-store' }`。
- 公司资料、历史财务、公告等低频数据应按业务周期缓存，不要求每 5 秒重新请求。
- 现有项目刷新频率继续以 `PRD.md` 的真实生产基线为准；新增雷达频率以 V5.0 规划书为上限建议，并必须经过数据源限频、单轮耗时和资源占用验证。
- 不得把行情、行业、ETF、龙头、公告、普通新闻和公司财务强行统一成一个刷新频率。
- 高频任务必须防重入、限制并发，记录开始、成功、失败和耗时；非交易时段应自动降频或停止不必要计算。
- 每一种数据必须区分“原始数据时间”“本系统抓取时间”和“页面渲染时间”，不得把浏览器当前时间冒充数据源更新时间。
- 请求失败时保留明确错误状态；不得静默展示上一只股票或过期数据。

## 6. 数据真实性与可追溯性底线

生产功能中的所有数据必须属于以下四类之一：

1. **原始数据**：来自明确、可追溯的公开数据源。
2. **计算数据**：由真实原始数据按明确公式计算，需说明口径。
3. **AI 判断**：明确标注为模型推断，不得伪装成事实或新闻。
4. **缺失/失败**：返回 `null`、空数组、“暂无数据”或“抓取失败”。

强制规则：

- 生产接口和生产页面绝对禁止 Mock 新闻、虚构行情、硬编码财务数字、模拟资金流或为了填满界面而补假数据。
- 测试可以使用 Mock/Fixture，但只能存在于测试目录，不能进入生产返回值。
- 数据缺失不得用 `0` 代替；真正的 0 与缺失值必须可区分。
- 不得用净利润冒充扣非净利润，不得用当日涨跌幅推算 20 日涨跌幅、量比或资金流。
- 每条事实性新闻、政策、公告应尽量提供原文链接、来源和发布时间；没有可靠原文时不得标记“已验证”。
- AI 没有真实联网或检索能力时，不得在 Prompt 或页面中宣称“自动联网”。没有可靠证据时必须明确说“暂无法确认”。
- AI 未运行、调用失败、未配置密钥、股票未加入监测是四种不同状态，不能统一伪装成 50 分。
- V5.0 的市场环境、行业主线、ETF 观察池和龙头规则状态必须由真实数据、确定性规则、版本化配置和状态机决定。
- AI 只能整理线索、解释证据、反证、风险和失效条件，不得直接改变正式状态或补齐缺失字段。
- 主营与催化的正式关联必须来自可追溯的官方分类、公司披露或人工审核的版本化映射；AI 生成的关联只能作为待验证线索。
- 新闻、量比、龙虎榜、单一资金流或单一媒体观点不得单独决定龙头状态。
- 每个参与正式计算的新字段必须定义主源、备用源、数据时间、最大延迟、覆盖率、缺失语义、冲突策略和是否参与本轮评分。
- 通俗总结保持专业，禁止出现“【小学生大白话】”等幼稚标题；前端需过滤此类意外前缀。

## 7. 接口、权限与密钥安全

- `.env`、`.env.*`、数据库、日志、虚拟环境、构建产物不得提交到 Git；仓库只允许提交不含真实值的 `.env.example`。
- `NEXT_PUBLIC_*` 会暴露给浏览器，绝对不能保存密钥或服务端访问凭证。
- 发现密钥曾进入 Git 时，先停止推送，提醒用户轮换密钥；清理历史和强制推送必须单独确认。
- 生产后端不得使用 `allow_origins=["*"]` 暴露写接口；只允许明确的前端域名和本地开发地址。
- 当前用户已明确接受单人自用系统的公开网址访问风险；登录、服务端会话、CSRF、用户权限和基于用户/会话的访问限流暂缓实施，且不再作为 V5.0 本地开发、影子运行或正式启用的前置门槛。只有用户以后明确要求时才重新纳入实施范围。
- 当前决定不等于允许泄露密钥：后端 Token 继续只由 Next.js 服务端代理持有，真实密钥不得发送到浏览器；直接访问后端敏感接口仍保留现有 Token 校验。
- 管理型全市场重算、历史回放和雷达 AI 手动调用继续由服务端功能开关控制并默认关闭；API 模型费用继续依赖用户已设置的供应商金额上限。未来若用户要求恢复访问保护，再补登录、会话、CSRF、权限、限流和相关测试。
- 前端不能直接保存后端私密 Token；需要私密凭证时，应通过服务端代理或正式登录体系转发。
- 错误响应不得泄露密钥、内部路径、完整堆栈或上游敏感信息。

## 8. 前后端数据契约

- 自选列表始终同时保存 `stockCode` 与 `stockName`。
- 接口字段名和缺失语义必须稳定；正常与失败分支不能返回不同名称的数据结构。
- 新增或修改接口时，优先使用 Pydantic/TypeScript 类型保持前后端一致。
- 金额、百分比、日期和股票代码在进入 UI 前统一格式化，不在多个组件中重复猜测数据格式。
- API 地址和请求公共逻辑逐步集中管理，不在多个页面重复硬编码临时 ngrok 地址。

## 9. 数据库与定时任务

- SQLite 数据库属于运行数据，不提交 Git，不在未备份时修改或删除。
- 修改表结构前必须备份，并提供迁移、结果核对和回退路径；禁止直接覆盖用户现有数据库。
- V5.0 第一阶段默认保留现有 SQLite；只有瓶颈得到真实验证、迁移方案和回退验证完成且用户确认后，才允许迁移其他数据库。
- 不得默认把全市场逐证券分钟级原始快照全部写入 SQLite；实施前必须估算日增量、保留周期、磁盘增长、查询时间和备份成本。
- 新数据库、新表或新存储接管正式业务前必须完成测试、数据核对和回滚验证；旧数据库、旧接口和提醒历史在新实现验收前保持可用。
- 浏览器缓存只能作为本地缓存，不能与后端数据库相互全量覆盖导致“最后写入者覆盖全部数据”。
- 定时任务必须记录开始、成功、失败和耗时；同一任务不得因多 worker 重复运行。
- 交易状态必须区分 A 股、港股、午间休市、周末和法定节假日；交易日历不得永久写死在单一年份。

## 10. 仓库卫生

- 不提交 `backend/venv`、`node_modules`、`.next`、日志、SQLite、`.DS_Store`、隧道运行文件和本机缓存。
- `patch_*.py`、`fix_*.py`、临时测试脚本、旧数据库和本地工具只有在确认无用途后才能删除。

- 后端依赖必须通过 `requirements.txt` 或 `pyproject.toml` 可复现；README 必须记录真实启动和验证方式。

## 11. 测试与验收

- 修复 bug 或改变行为时，优先先写能复现问题的失败测试，再做最小实现，再验证测试通过。
- 前端最低检查：相关测试、ESLint、TypeScript、Next.js build；涉及 UI 时补桌面端 16:9 页面检查。
- 后端最低检查：相关自动测试、核心模块语法/导入检查、关键接口的成功/缺失/失败分支。
- 数据类修改至少覆盖：真实 0、缺失值、上游失败、无效股票、A 股、港股。
- 部署前必须重新检查 Git diff，确认没有 `.env`、数据库、日志、密钥和无关文件。
- 不能运行的检查必须说明原因，不用“应该可以”代替证据。
- V5.0 不得破坏现有后端完整测试、ESLint、TypeScript、Next.js build 和现有生产页面。
- 现有提醒历史回放的“20 条、10 个日期”与雷达“至少 20 个交易日影子运行”是两个独立门槛，不能互相替代。
- 新增雷达必须覆盖空榜、缺失、过期、数据源失败、同股状态唯一、状态升级/降级/移出、阈值滞回、任务防重入、功能开关和 AI 不修改状态等测试；登录、CSRF、权限和用户限流测试只在用户以后启用访问保护时补充。
