# 1. 升级目标

现有系统已经完成“少量重点证券的准实时监测、提醒和证据解释”。

V5.0新增的能力是：

> 从“用户先选股票、系统持续监测”，升级为“系统先识别市场主线、行业ETF和龙头梯队，再让用户决定是否加入监测”。

升级后形成两条互补链路。

## 1.1 现有链路：重点证券监测

```text
用户选择证券
→ 加入监测列表
→ 行情、公告、行业和风险规则检查
→ 提醒入库与送达
→ AI解释
```

## 1.2 新增链路：全市场主线发现

```text
全市场和行业数据
→ 市场环境判断
→ 主线行业识别
→ 行业ETF筛选
→ 股票硬门槛
→ 预备龙头
→ 候选龙头
→ 已确认龙头
→ 证据、反证、风险和失效条件
→ 用户选择是否加入现有监测列表
```

新链路不得自动下单，也不得自动把股票加入监测列表。

# 4. 核心输出规则

## 4.1 四类核心输出

| 类别 | 最大数量 | 允许空榜 |
|---|---:|---:|
| 行业ETF候选 | 5 | 是 |
| 预备龙头 | 5 | 是 |
| 候选龙头 | 5 | 是 |
| 已确认龙头 | 5 | 是 |

不得为了填满5只降低门槛。

## 4.2 龙头状态路径

升级：

```text
普通股票
→ 预备龙头
→ 候选龙头
→ 已确认龙头
```

恶化：

```text
已确认龙头
→ 候选龙头
→ 预备龙头
→ 移出
```

同一股票只能处于一个状态。

## 4.3 ETF周期

### 趋势轮动

```text
5～20个交易日
```

### 中期跟踪

```text
1～6个月
```

ETF状态不等于买入建议。

## 4.4 股票周期

龙头识别主要服务：

```text
1～20个交易日的短线至波段市场地位
```

已确认龙头不代表当前位置适合追涨。

“预备龙头、候选龙头、已确认龙头”均为规则识别的市场地位状态，不是买入、卖出、仓位或收益结论；页面必须在状态附近持续展示该说明。

## 4.5 观察对象真实性分层

“观察对象真实”必须拆成三层，不得混为一个承诺：

1. **事实真实性**：证券、ETF、指数、行业、公告和原始行情必须真实存在，并保存来源、原始时间、抓取时间和完整度；无法验证时标记缺失或失败，不生成事实。
2. **计算正确性**：所有特征、分数、排名和状态迁移必须使用版本化公式和同一批次输入，可重复计算，并由自动测试覆盖真实0、缺失、过期和异常值。
3. **判断有效性**：主线行业、ETF观察池和三级龙头属于确定性规则基于真实数据形成的研究判断，不是客观事实本身，仍可能出现误报、漏报或阶段判断错误，必须通过独立黄金样本、严格时间点回放和影子运行持续验证。

正式页面必须明确区分：

- 已验证事实；
- 规则计算结果；
- AI解释；
- 待验证线索；
- 缺失或失败；
- 主要反证和首次否决原因。

任何观察对象都必须能回答“为什么出现、使用了哪些数据、哪些字段缺失、什么会使判断失效”。系统承诺的是不造假、可追溯和可复算，不承诺每次研究判断都正确，更不得把观察状态表达成投资结论。

# 5. 股票池和硬过滤

## 5.1 第一阶段雷达范围

纳入：

- 沪市主板；
- 深市主板；
- 创业板；
- 科创板。

不纳入雷达扫描，但不影响现有单股监测：

- 港股；
- 北交所。

## 5.2 全量扫描与分层计算

龙头识别必须从第一阶段范围内的全部股票开始，不得只扫描当前监测列表、人工候选、历史强势股或媒体热门股。

每个交易日开始前，依据带生效日期的交易所证券名单建立当日全量股票池；盘中每轮计算使用统一的 `as_of` 时间、批次编号和规则版本，禁止把不同抓取时点的数据静默拼成同一横截面排名。

分层计算：

1. **全量证券建册层**：覆盖沪市主板、深市主板、创业板和科创板全部证券，保存上市日期、交易状态、板块、行业、风险标记和当日是否可参与雷达。
2. **统一硬过滤层**：对全部证券执行 ST、退市、停牌、新股、风险、流动性和数据完整度过滤，并保存每只证券的纳入或排除原因。
3. **全体轻量计算层**：每个盘中雷达周期对全部合格证券计算行情、成交、换手、相对行业强度、相对大盘强度和数据质量等可批量获得的轻量指标。
4. **行业内深度计算层**：只对进入观察、启动或确认阶段的行业及其中排名靠前的股票，进一步计算领先性、持续性、抗跌、分歧、回流和行业带动性。
5. **候选证据核验层**：只对通过硬门槛的少量候选加载公告、主营、催化、风险和历史证据；AI只能在确定性状态产生后解释结果。

每轮运行必须记录：

- 股票池总数；
- 合格证券数；
- 实际扫描成功数；
- 失败、过期和缺失数；
- 各排除原因数量；
- 各层进入和淘汰数量；
- 批次开始、结束、数据截止时间和耗时。

扫描覆盖率未达到该规则版本的最低要求时，本轮不得生成新的正式行业或龙头状态，只能保留上一成功状态并明确标记过期。全量扫描不等于对全部股票逐一抓取公告、财务或调用AI，也不等于把全部分钟原始数据写入SQLite。

## 5.3 排除范围

- ST；
- *ST；
- 退市整理；
- 停牌；
- 明确重大退市风险；
- 重大立案调查且风险未解除；
- 数据严重缺失；
- 极低流动性；
- 无法正常成交的连续一字板；
- 交易所明确异常交易状态。

## 5.4 新股

- 上市后的前5个交易日：不进入雷达展示；
- 上市后的第6～10个交易日：最多进入预备龙头，置信等级封顶；
- 上市超过10个交易日：按普通规则。

## 5.5 小盘股

不按市值一刀切，组合判断：

- 流通市值；
- 5日和20日中位成交额；
- 换手率；
- 买卖价差；
- 盘口深度；
- 冲高回落；
- 监管和基本面风险。

## 5.6 盘口与高频字段准入

- 盘口深度、精确买卖价差、炸板率和最高连板等字段，只有在存在稳定、合法、可追溯且带数据时间的真实来源时才能参与正式计算。
- 第一阶段没有可靠盘口来源时，盘口深度不得作为硬门槛，也不得用成交额、换手率或其他字段伪造替代。
- 缺失字段必须标记为缺失，并根据版本化规则决定降级计算或停止判断，不能按0处理。

## 5.7 分板块交易规则与证券生命周期

主板、创业板、科创板、风险警示股票、新股、重新上市和退市整理证券的交易制度不同。雷达必须使用带生效日期的版本化规则表，不能用一套固定涨跌幅阈值处理全部证券。

规则表至少保存：

- 交易所、板块和证券类型；
- 证券代码、简称、上市日期、退市日期和代码变更；
- ST、*ST、退市整理、停牌、重新上市和异常交易状态；
- 当日适用的涨跌幅限制比例；
- 是否处于无涨跌幅限制交易日；
- 集合竞价、连续竞价和盘后交易阶段；
- 规则来源、发布日期、生效日期和失效日期。

涨停、跌停、炸板和异常波动必须按该证券当日实际适用规则计算，不能只判断固定的10%或20%。交易所规则发生变化时，新旧历史按各自生效版本计算，不得用当前规则覆盖过去。

## 5.8 价格、复权、公司行为与股本口径

不同用途使用不同价格口径：

- 涨跌停、成交价格、不可交易状态和订单价格判断使用当时未复权价格；
- 趋势、历史涨跌和相对强度使用同一复权口径的连续价格序列；
- 价格收益与包含现金分红的总回报分开；
- 换手率使用当时生效的自由流通股本或可交易股本，不使用当前股本回算过去；
- 市值、行业权重和指数贡献必须保存所用总股本、流通股本和权重日期。

必须记录除权、除息、送股、转增、配股、拆股、缩股、增发、回购注销、合并、分拆和证券代码变更等公司行为及生效日期。公司行为数据缺失时，涉及趋势跳变、换手和相对强度的判断停止或降级，不得把除权缺口误判成价格崩跌，也不得把股本变化误判成成交异常。

# 7. 主线行业引擎

## 7.1 核心问题

- 行业是否整体走强；
- 是否只有一只股票独涨；
- 成交额是否放大；
- 上涨是否扩散；
- 是否持续；
- 分歧后是否回流；
- 是否存在可信催化。

## 7.2 初始权重

| 模块 | 权重 |
|---|---:|
| 相对大盘强度 | 25 |
| 行业成交额和放量 | 20 |
| 上涨广度 | 15 |
| 持续性和回流 | 15 |
| 涨停和强势股扩散 | 10 |
| 政策、产业和业绩催化 | 10 |
| ETF辅助确认 | 5 |

权重必须配置化，不写死在多个文件。

## 7.3 双口径继承

现有PRD已经明确：

- 同花顺行业涨跌和资金；
- 新浪行业成分加腾讯实时行情。

V5.0继续分开计算，不混成同一统计集合。

可以在行业评分中使用两个独立维度，但必须保存：

- 来源；
- 样本范围；
- 数据时间；
- 完整度；
- 是否参与本轮评分。

## 7.4 评分计算合同

初始权重只表示产品方向，不等于可直接编码的公式。阶段3开始前，每个评分维度必须形成一份版本化计算合同，至少定义：

- 原始字段和单位；
- 主数据源、备用源和数据时间；
- 统计窗口和最小样本量；
- 归一化公式和上下界；
- 缺失、过期、冲突和异常值处理；
- 最低数据完整度；
- 是否允许降级计算；
- 进入、保持和退出阶段的阈值；
- 本轮是否参与评分及原因。

计算合同经测试和用户确认后才能成为正式规则版本。不得把同一个阈值或权重写死在多个文件中。

## 7.5 主行业、主题与分类版本

阶段0必须确定一套版本化的主行业分类，作为全市场横截面排名和历史回放的统一骨架。主行业分类与同花顺行业、新浪行业、指数行业和主题标签分别保存，不得互相冒充。

分类要求：

- 每只股票在同一生效时点只有一个主行业；
- 主题、概念和产业链标签允许多对多；
- 主行业、二级行业、细分行业和主题分别建模；
- 保存分类来源、分类版本、公布日期、生效日期和失效日期；
- 行业变更使用当时公开信息和实际生效日；
- 无法唯一映射时标记“行业映射未确认”，不强行归类；
- 历史回放只能使用当时有效分类，不能用当前分类覆盖过去。

行业强度至少同时观察：

- 市值或自由流通市值加权表现；
- 等权表现；
- 上涨广度；
- 成交额扩散；
- 前几大权重股贡献；
- 剔除第一大贡献股后的行业表现。

这样可以区分“整个行业走强”和“单只大权重股拉动”。主题主线必须单独标明为主题，不得在页面上伪装成稳定行业分类。

# 8. 行业ETF引擎

## 8.1 数据要求

ETF引擎必须先建立全部国内上市ETF的完整名册，再按宽基、行业、主题、策略、债券、商品、货币和跨境等类型分类。主线雷达从全量行业和主题ETF中筛选观察对象；宽基ETF主要作为市场和相对基准，其他类别默认不进入行业ETF观察池。

同一ETF可能跨多个行业或主题时，必须保存对应指数、成分权重、行业暴露比例和映射版本，不得为了方便强行只归入一个行业。

同一行业多只ETF比较：

- 基金规模；
- 日均成交额；
- 买卖价差；
- 折溢价稳定；
- 跟踪误差；
- 标的指数相关性；
- 份额变化；
- 上市和清盘风险；
- 指数成分和权重。

刷新分层：

- ETF价格、成交额和可获得的价差：按盘中雷达周期更新；
- 基金规模、份额、折溢价和跟踪数据：按真实发布频率更新；
- 标的指数成分和权重：按指数公司或基金披露的生效日期更新；
- ETF成分股盘中表现：优先复用同一批次的全市场股票扫描结果，不重复抓取；
- 不得每分钟重复下载低频持仓、指数成分或基金定期报告。

每轮ETF扫描必须记录ETF名册总数、行业/主题ETF数量、成功计算数、过期数、数据缺失数、排除原因和覆盖率。覆盖不足时允许空榜，不得用旧ETF或熟悉ETF补位。

## 8.2 官方来源优先

新增：

- 上交所和深交所ETF产品信息；
- 基金管理人官网；
- ETF公告；
- 指数公司；
- 基金定期报告。

现有聚合数据可以做发现和补充，但ETF正式字段优先采用官方来源。

## 8.3 输出

- ETF代码；
- ETF名称；
- 行业；
- 周期类型；
- 规则匹配度；
- 流动性；
- 规模；
- 跟踪质量；
- 份额变化；
- 主要证据；
- 主要风险；
- 降级和退出条件。

行业ETF输出属于规则观察池。不得使用“推荐买入”“配置优先级”“永久持有”等投资建议式文案。

## 8.4 ETF产品与标的指数尽调

ETF进入正式观察池前必须完成产品层和指数层双重核验。

产品层至少记录：

- 被动指数ETF或主动ETF；
- 交易所、基金管理人、托管人、上市日期和基金状态；
- 基金规模、份额、日均成交额、可获得的买卖价差和折溢价；
- 申购赎回清单、现金替代和暂停申购赎回状态；
- IOPV来源和更新时间，并明确IOPV只是参考值，不等于真实基金净值；
- 基金合同终止、清盘、合并、转型和终止上市风险；
- 同一产品的场内外份额、联接基金和重复代码关系，避免重复统计。

指数层至少记录：

- 标的指数、指数公司和编制方案版本；
- 样本空间、选样规则、调样频率和实际生效日；
- 加权方式、单一成分上限和前十大成分集中度；
- 行业纯度、主题暴露和跨行业权重；
- 与同类指数及ETF的成分重合度；
- 跟踪误差和跟踪差异，二者不得混为一个指标；
- 指数成分、权重和公司行为使用当时有效版本。

主动ETF没有固定标的指数，必须单列。V5.0第一阶段主动ETF可以进入产品名册和健康监测，但不得与被动行业指数ETF使用同一排名公式，也不得在独立规则、数据和回放未验收前进入正式行业ETF观察池。

ETF份额增加或减少不直接等于二级市场“资金净流入”或“资金净流出”。页面和规则必须写成份额变化、申购赎回相关变化或来源原始口径，只有数据源明确给出并可验证时才展示资金口径。

同一指数有多只ETF时先归为同一指数产品组，再比较规模、流动性、跟踪质量和运行风险；不得让同一指数的多个重复ETF占满5个观察名额。

# 9. 龙头引擎

## 9.1 判断顺序

```text
行业进入主线
→ 股票硬门槛
→ 领先性和相对强度
→ 成交和流动性
→ 持续性、抗跌和回流
→ 主营与催化关联
→ 风险过滤
→ 状态机
```

## 9.2 预备龙头

行业已开始增强，个股出现领先，但尚未充分验证：

- 行业进入观察、启动或确认；
- 相对行业强度靠前；
- 成交额和换手改善；
- 较早异动、突破或涨停；
- 主营和催化有关联；
- 无重大风险；
- 尚未完成分歧和回流验证。

## 9.3 候选龙头

- 行业主线较明确；
- 连续强于行业和大盘；
- 流动性合格；
- 市场地位靠前；
- 持续性、抗跌或回流证据成立；
- 催化和主营关联明确；
- 不存在明显高位失效。

## 9.4 已确认龙头

- 行业已经明确；
- 行业启动时处于前列；
- 经历分歧或回调后保持地位；
- 行业回流时率先修复；
- 具有代表性或带动性；
- 成交和流动性持续合格；
- 无严重滞涨和重大风险。

## 9.5 硬门槛

### 行业门槛

- 行业进入观察、启动或确认；
- 不是单股独涨；
- 行业成交额合格；
- 有扩散、持续或回流；
- 催化来源可靠；
- 数据完整。

### 个股门槛

- 相对行业强度处于前列；
- 成交额合格；
- 换手和成交结构正常；
- 主营和催化直接或高度相关；
- 不属于排除范围；
- 无高位巨量滞涨；
- 无连续冲高回落；
- 领先性、持续性和抗跌性至少两项成立。

“主营和催化关联”只有在官方行业分类、公司公告、年报主营、公司IR或人工审核的版本化映射支持时才能参与正式门槛。AI和新闻标题只能生成待验证线索；无法验证时必须显示“关联未确认”，不得自动升级状态。

## 9.6 初始评分

| 模块 | 权重 |
|---|---:|
| 行业主线强度 | 25 |
| 市场地位和领先性 | 25 |
| 相对强度和持续性 | 20 |
| 成交、换手和流动性 | 15 |
| 催化和主营关联 | 10 |
| 量比、龙虎榜等辅助 | 5 |

辅助资金指标总权重不得超过5%，不能绕过硬门槛。

## 9.7 风险和失效

风险标签：

- 短期涨幅过高；
- 高换手；
- 高位放量；
- 冲高回落；
- 行业持续性不足；
- 催化关联弱；
- 个股弱于行业；
- 数据不足；
- 减持；
- 监管；
- 业绩风险。

降级或移出：

- 行业强度下降；
- 个股连续弱于行业；
- 高位放量但无法创新高；
- 连续冲高回落；
- 重大风险公告；
- 数据源异常；
- 主营关联被证伪；
- 流动性恶化。

## 9.8 状态机正式合同

阶段6开始前必须为每个状态和每条迁移边定义：

- 进入分数和硬门槛；
- 保持分数；
- 降级和移出条件；
- 连续满足次数；
- 最短保持时间；
- 阈值滞回区间，避免在边界反复升降级；
- 降级后的冷却期和重新进入条件；
- 数据缺失、过期和数据源失败时是保持、降级还是停止判断；
- 市场环境和行业状态变化时的联动规则；
- 同一证券只能有一个状态的数据库约束与测试。

上述内容必须进入单一版本化配置，并随每次雷达快照保存规则版本。没有正式合同前，只允许生成研究性特征，不得生成正式三级状态。

## 9.9 龙头角色、业务暴露与首次否决

“价格最强、成交核心、行业代表和产业基本面核心”可能不是同一只股票。系统内部必须分别计算并展示角色证据，不能把涨幅第一直接等同于行业龙头。

龙头判断至少拆分：

- **市场领先性**：启动顺序、相对强度、分歧后修复和持续性；
- **行业贡献度**：行业成交额占比、行业权重、上涨贡献和带动扩散；
- **业务暴露证明**：主营收入、产品、订单、产能、客户或公司正式披露与行业催化的可追溯联系；
- **流动性与可交易性**：成交额、换手、价差、停牌、涨跌停和一字板；
- **风险与失效**：减持、解禁、监管、调查、诉讼、业绩、审计和业务关联被证伪。

业务暴露只有新闻提及、主题标签或AI推断而没有公司披露和可追溯材料时，标记“业务暴露未证实”，最多进入预备观察，不能进入候选或已确认状态。

每个候选必须显示“首次否决原因”：当前最可能使其被淘汰的证据缺口或风险。单日涨幅、单次涨停、量比、龙虎榜、单日资金或媒体热度只能作为辅助字段，不能绕过行业门槛、业务暴露和状态机。

评分必须检查小市值、高换手、连续涨停和流动性不足带来的偏置。可交易性不足的股票可以作为市场现象记录，但不得因价格涨幅高而自动获得更高正式状态。

# 10. 证据、反证和独立雷达AI

## 10.1 正式判断

正式状态只能由：

```text
真实数据
+ 硬门槛
+ 配置化评分
+ 状态机
```

决定。

## 10.2 雷达AI职责

AI可以：

- 解释市场环境；
- 解释行业催化；
- 解释ETF和龙头进入原因；
- 整理证据和反证；
- 识别行业、公司和产业链实体；
- 生成通俗总结。

AI不能：

- 直接选出龙头；
- 修改状态；
- 绕过门槛；
- 编造行情；
- 单凭新闻标题升级股票；
- 给出确定买卖和收益承诺。

雷达AI的业务名称统一为“雷达AI解读”。它只服务于主线雷达，不替代现有股票详情页的单股事件AI。

## 10.3 与现有单股AI严格隔离

当前生产系统已经存在单股事件AI：

- 后端入口：`backend/ai_analysis.py`；
- 当前提示词版本：`evidence-v3`；
- 当前接口：`/api/stock/ai_attribution/{symbol}` 和单股AI历史接口；
- 当前历史表：`ai_analysis_history`；
- 当前前端：股票详情中的 `AiAttributionTab`；
- 当前任务：只针对监测列表，按既有定时复盘和事件触发运行。

V5.0雷达AI不得通过在上述模块中增加业务分支实现。两套AI必须在以下方面完全隔离：

- 业务代码；
- API路由；
- 输入输出合同；
- 提示词和提示词版本；
- 历史记录和数据库表；
- 证据指纹和缓存；
- 自动任务、任务锁和健康状态；
- 使用配额、失败熔断和审计；
- 前端组件和页面状态；
- 模型配置和功能开关。

允许共用的范围仅限无状态、无提示词、无缓存、无业务数据的底层OpenAI兼容HTTP客户端、超时工具和通用脱敏工具。不得为了共用客户端重构现有单股AI；如果抽取公共层会扩大阶段8的修改范围，雷达AI先使用自己的轻量客户端。

建议雷达AI使用独立服务端配置：

```text
RADAR_LLM_API_KEY
RADAR_LLM_BASE_URL
RADAR_LLM_MODEL
```

这些变量不得使用 `NEXT_PUBLIC_*`，不得发送到浏览器。即使雷达AI与现有单股AI实际连接同一家模型供应商，也必须能分别启停、分别限额和分别判断故障。新增或修改生产环境变量仍需用户单独授权。

## 10.4 四类独立分析任务

雷达AI不使用一个通用大提示词包办全部任务，至少拆分为四类版本化合同：

| 分析任务 | 建议提示词版本 | 分析对象 | 主要输出 |
|---|---|---|---|
| 市场环境 | `radar-market-v1` | 单个合格雷达批次 | 市场强弱、主线集中度、扩散或退潮、证据缺口 |
| 行业主线 | `radar-sector-v1` | 单个行业及其历史状态 | 进入原因、阶段、催化、反证、降级和退出条件 |
| 行业ETF | `radar-etf-v1` | 单只ETF及同指数产品组 | 指数与成分、行业纯度、流动性、跟踪差异和产品风险 |
| 龙头梯队 | `radar-leader-v1` | 单只预备、候选或已确认股票 | 市场领先性、行业贡献、业务暴露、可交易性、首次否决原因和失效条件 |

ETF分析必须明确区分ETF份额变化、成交活跃、折溢价、IOPV、申购赎回清单、跟踪误差和跟踪差异，不得把其中任一字段自动改写成“资金净流入”。龙头分析不得因为涨幅第一、新闻热度或AI判断绕过正式门槛。

## 10.5 不可变输入合同

雷达AI只能读取确定性雷达引擎完成计算后冻结的证据包。建议统一输入合同至少包含：

```text
radarRunId
batchId
asOf
ruleVersion
coverage
scopeType
scopeId
formalState
formalScoreBreakdown
stateHistory
evidence
counterEvidence
firstRejectionReason
unknowns
sourceCatalog
dataCompleteness
```

强制要求：

- `radarRunId`、`asOf`、规则版本和扫描覆盖率不可缺失；
- `formalState` 和正式评分只允许作为只读输入，AI无权返回修改后的状态；
- 每条事实证据和反证都必须具有稳定来源编号、原始数据时间、抓取时间和可用状态；
- 待验证新闻、主题标签和AI实体识别必须与正式证据分开；
- 输入覆盖率未达到该规则版本最低要求时，不生成新的正式雷达AI解读；
- 证据包冻结后不得在同一分析运行中读取“最新值”替换其中字段，避免混用不同时点。

雷达AI不能自行扫描全市场，也不能逐只股票自由抓取新闻。全市场发现由确定性扫描层完成，AI只分析通过门槛的少量对象及其冻结证据包。

## 10.6 结构化输出合同

雷达AI输出至少包含：

```text
analysisStatus
confirmedFacts
inferences
unknowns
counterEvidence
conditionalScenarios
plainEnglishSummary
sourceIds
invalidatingConditions
promptVersion
model
analysisAt
durationMs
usage
```

输出约束：

- `analysisStatus` 必须区分成功、未运行、未配置、证据不足和调用失败；
- 已确认事实、模型推断、暂无法确认和待验证线索分开返回；
- `sourceIds` 只能引用本次冻结证据包中的来源编号；
- 后端必须拒绝不存在的来源编号、无法解析的JSON和越权状态字段；
- AI输出不得包含新的正式分数、排名、优先级、行业状态、ETF状态或龙头状态；
- 条件情景只能说明改善、延续、恶化和失效条件，不得给出目标价、收益承诺或确定性涨跌；
- 模型解释失败时保留规则结果，前端明确显示失败状态，不用旧AI结果或默认分数填充。

## 10.7 触发、复用、费用和安全

推荐采用“合格批次自动概览 + 状态变化自动解释 + 详情手动核对”的混合策略：

- 每个达到覆盖门槛且证据发生变化的雷达批次，最多生成一次市场环境解读；
- 行业进入、阶段变化、降级或退出时生成行业解读；
- ETF进入、周期变化或移出观察池时生成ETF解读；
- 龙头升级、降级、移出或首次否决原因变化时生成龙头解读；
- 用户打开详情后可手动要求重新核对；当前单人自用模式不要求登录、权限、CSRF和用户限流，但必须由独立服务端功能开关控制，默认关闭，并受模型供应商金额上限约束；
- 不为全市场每只股票逐轮调用AI，不为没有状态变化的对象重复调用AI。

雷达AI复用键至少包含：

```text
radarRunId + scopeType + scopeId + evidenceFingerprint + promptVersion + model
```

同一复用键只能产生一次有效分析任务。证据指纹未变化时复用雷达AI自己的成功结果；不得复用旧单股AI缓存。证据、规则版本、提示词版本或模型版本任一变化时重新分析。

雷达AI任务必须：

- 使用独立任务锁和独立健康状态；
- 不阻塞FastAPI主事件循环；
- 单次超时，不进行SDK无限重试；
- 记录耗时、用量、错误分类和审计信息，但不记录密钥和完整上游敏感响应；
- 达到日内或会话配额后停止新调用，规则计算和雷达页面继续正常工作；
- 功能关闭或模型故障时不影响现有单股AI、提醒、邮件和雷达正式状态。

## 10.8 每个候选必须显示

- 证据；
- 反证；
- 风险；
- 数据来源；
- 数据时间；
- 数据完整度；
- 规则版本；
- AI模型和提示词版本；
- 失效条件。

# 11. 数据源整合与冲突处理

## 11.1 现有数据源处理矩阵

| 当前来源 | 当前用途 | V5.0处理 |
|---|---|---|
| 腾讯财经 | 行情、K线、成分实时行情 | 保留，作为主要行情源 |
| 新浪个股资金 | A股个股主力净额 | 保留，只做个股辅助维度 |
| 同花顺行业资金 | 行业涨跌、资金和排名 | 保留，独立行业资金口径 |
| 新浪行业成分 | 行业成分 | 保留，与腾讯行情组合 |
| 东方财富 | 公司和财务 | 保留 |
| 巨潮资讯 | A股官方公告 | 保留并强化正式证据地位 |
| 多源新闻和政策 | 资讯发现 | 保留，按S/A/B/C使用 |
| 上交所/港交所日历 | 交易状态 | 保留 |

## 11.2 新增来源

- 上交所和深交所证券名单、交易规则与证券状态；
- 上交所和深交所ETF官方信息；
- 基金管理人；
- 指数公司；
- 指数编制方案、成分、权重和调样公告；
- 公司行为、分红、股本和证券代码变更公告；
- 政府和监管政策；
- 公司IR；
- 行业协会；
- 海外关键公司IR和监管；
- 必要的高时效财经快讯。

## 11.3 冲突处理原则

1. 不静默替换现有稳定数据源；
2. 同字段多源冲突时保留原始值和来源；
3. 先按用途分工，不追求一个来源包办全部；
4. 官方源负责确认，快讯负责发现；
5. 不同统计口径不直接相加、相减或平均；
6. 数据源失败时明确降级；
7. 新源必须先影子运行；
8. 新源稳定性未验证前不参与正式状态；
9. 每个正式字段定义主源、备用源、最大延迟和冲突策略。

## 11.4 来源等级

- S：政府、监管、交易所、公司公告、公司IR、基金公告；
- A：行业协会、权威机构、主流财经媒体；
- B：普通财经和行业网站；
- C：社交媒体、论坛和传闻。

C级只能作为待验证线索。

## 11.5 字段级数据源合同

阶段0必须为每个准备进入正式计算或展示的字段建立数据源合同：

| 合同项 | 要求 |
|---|---|
| 字段定义 | 名称、单位、统计口径和适用证券范围 |
| 主源与备用源 | 明确到实际接口或采集模块 |
| 原始时间 | 数据源时间，不使用抓取时间冒充 |
| 最大延迟 | 超过后标记过期，不继续当作实时数据 |
| 覆盖率 | 记录市场、证券和交易日覆盖情况 |
| 缺失语义 | 区分真实0、缺失、失败和不适用 |
| 冲突策略 | 保留原始值，不跨口径直接混算 |
| 调用限制 | 频率、并发、超时、重试和来源使用约束 |
| 正式准入 | 影子稳定性未通过前不得改变正式状态 |

当前来源未明确覆盖的涨停/跌停、炸板率、最高连板、历史行业成分、ETF份额、跟踪误差、精确价差和盘口深度，必须先完成可行性验证。无法取得真实可靠数据时，应删除对应正式依赖或降级为不参与判断，不能寻找相似字段冒充。

# 12. 数据库和技术架构策略

## 12.1 第一阶段不强制迁移数据库

现有SQLite承载真实生产数据。

V5.0第一阶段：

- 保留SQLite；
- 通过备份和迁移脚本新增雷达表；
- 不直接迁移PostgreSQL；
- 不影响现有提醒和AI历史。

## 12.2 建议新增表

名称可根据现有代码规范调整：

- `radar_runs`
- `market_environment_snapshots`
- `sector_strength_snapshots`
- `sector_state_history`
- `etf_candidates`
- `leader_candidates`
- `leader_state_history`
- `candidate_evidence`
- `radar_source_status`
- `radar_replay_runs`
- `radar_rule_versions`
- `security_master_history`
- `market_rule_versions`
- `corporate_actions`
- `industry_classification_history`
- `industry_theme_mappings`
- `etf_product_registry`
- `etf_index_version_history`
- `etf_constituent_history`
- `radar_gold_sample_labels`
- `radar_ai_analysis_runs`
- `radar_ai_outputs`

实际表数量可以根据现有仓储模式合并，但证券生命周期、交易规则、公司行为、行业分类、ETF指数版本和黄金样本的历史有效期不能只保存在当前值字段中。

`radar_ai_analysis_runs` 保存任务身份、雷达批次、分析范围、证据指纹、提示词和模型版本、状态、耗时、用量及错误分类。`radar_ai_outputs` 保存结构化解释和实际引用来源。两表不得与现有 `ai_analysis_history` 合并，也不得通过股票代码把旧单股AI历史当作雷达AI结果。

## 12.3 可选分析存储

历史数据量增大后，再评估：

- Parquet：原始快照和回放输入；
- DuckDB：历史分析和回放查询；
- Polars：新增全市场批量计算。

规则：

- 新雷达批量计算可以优先使用Polars；
- 已有稳定pandas代码不强制重写；
- DuckDB不承担现有生产业务状态；
- PostgreSQL只有在多用户、多实例、云端化或SQLite瓶颈明确后再迁移；
- Redis、Dramatiq和pgvector不是第一阶段强制依赖。

### 12.3.1 第一阶段存储边界

- SQLite优先保存雷达运行记录、市场和行业聚合快照、候选结果、状态变化、证据、规则版本和数据源健康。
- 不得默认把全市场约数千只证券的逐分钟原始快照全部写入SQLite。
- 阶段0必须估算单轮证券数量、单轮字节数、每日运行次数、日增量、保留周期、磁盘增长、查询时间、备份时间和回放成本。
- 如果历史回放确实需要批量原始快照，应先给出分区、压缩、保留和删除策略，再单独评估Parquet或其他分析存储；新增依赖和正式迁移仍需用户授权。
- 任一存储方案都必须保留数据时间、抓取时间、来源、规则版本和完整度，不能只保存最终分数。

## 12.4 模块边界

在现有后端结构中新增独立雷达模块，至少包含：

- 市场环境；
- 股票池过滤；
- 行业特征；
- 行业状态；
- ETF候选；
- 龙头特征；
- 龙头评分；
- 龙头状态机；
- 证据；
- 回放；
- API；
- 调度任务。

必须沿用现有项目的路由、数据库、错误结构、日志和测试模式，不另起重复项目。

雷达AI在雷达模块内部保持独立边界，建议结构为：

```text
backend/radar/ai/
├── service.py
├── contracts.py
├── prompts.py
├── validator.py
├── repository.py
└── scheduler.py
```

职责要求：

- `service.py` 只负责编排冻结证据包、模型调用和结果状态；
- `contracts.py` 定义雷达AI输入输出合同，不导入旧单股AI响应模型；
- `prompts.py` 分别保存市场、行业、ETF和龙头提示词版本；
- `validator.py` 校验来源编号、字段类型、越权状态字段和禁止输出；
- `repository.py` 只访问雷达AI专用表；
- `scheduler.py` 只处理雷达AI触发、去重、任务锁、配额和健康记录。

前端新增雷达AI专用组件，不导入或复用 `AiAttributionTab` 的业务状态。可以复用纯展示基础组件，但不得因此共用请求、历史、缓存或数据类型。

# 13. API设计

建议新增：

```text
GET /api/radar/overview
GET /api/radar/market-environment
GET /api/radar/sectors
GET /api/radar/sectors/{sector_id}
GET /api/radar/etfs
GET /api/radar/etfs/{etf_code}
GET /api/radar/leaders?status=preliminary
GET /api/radar/leaders?status=candidate
GET /api/radar/leaders?status=confirmed
GET /api/radar/leaders/{symbol}
GET /api/radar/leaders/{symbol}/history
GET /api/radar/history
GET /api/radar/health
POST /api/radar/recalculate
POST /api/radar/replays
GET /api/radar/replays/{run_id}
```

雷达AI独立建议接口：

```text
GET  /api/radar/ai/overview
GET  /api/radar/ai/sectors/{sector_id}
GET  /api/radar/ai/etfs/{etf_code}
GET  /api/radar/ai/leaders/{symbol}
GET  /api/radar/ai/history?scope_type={scope_type}&scope_id={scope_id}
POST /api/radar/ai/analyze
```

雷达AI接口只读写雷达AI专用合同和专用表，不代理旧 `/api/stock/ai_*` 接口。所有返回值必须携带 `radarRunId`、`asOf`、`analysisStatus`、证据指纹、提示词版本和模型版本，防止页面把不同批次的解释混在一起。

写接口：

- 必须走现有Next.js服务端代理；
- 当前继续使用只由服务端代理持有的后端Token，不得把Token发送给浏览器；用户已接受公开代理访问风险，暂不建设登录、服务端会话、CSRF、用户权限和用户限流；
- 手动重算不能绕过任务锁；
- 生产模式不得允许普通浏览器无限触发全市场重算。

所有雷达页面和只读API当前可以沿用现有公开访问模式。`POST /api/radar/recalculate` 和 `POST /api/radar/replays` 必须由独立服务端功能开关控制并默认关闭；用户明确授权启用后仍需任务锁、调用审计和单实例约束。以后用户要求收紧访问时，再增加登录会话和管理员权限。

`POST /api/radar/ai/analyze` 由 `RADAR_AI_MANUAL_ENABLED` 单独控制，默认关闭；启用后继续受到模型供应商金额上限约束。当前不增加用户/会话限流；以后用户要求访问保护时再设计限额。自动任务同一复用键最多执行一次。

| 任务 | 频率 |

| 行业强度 | 1～3分钟 |
| ETF候选 | 3～5分钟 |

| 收盘快照 | 收盘后 |
| 历史回放 | 手动或盘后 |
| 规则版本归档 | 每次发布 |
| 雷达AI市场概览 | 合格雷达批次完成且证据变化后 |
| 雷达AI对象解释 | 行业、ETF或龙头状态变化后，或受保护的手动核对 |

以上频率是交易时段内的目标上限，不是未经验证的固定承诺。阶段0必须测量各数据源限频、单轮抓取耗时、单轮计算耗时、内存、CPU和磁盘写入；如果无法在下一周期前稳定完成，应降低频率、缩小输入或停止该字段参与正式状态。

约束：

- 高频任务 `max_instances=1`；
- 不重叠运行；
- 不阻塞FastAPI主事件循环；
- 同一交易时点只保存一份正式状态；
- 失败时保留上一成功状态并明确标记过期；
- 新任务必须进入现有健康接口。
- 午间休市、收盘后、周末和法定节假日按任务用途降频或停止，不得继续无意义地执行全市场分钟计算。
- 雷达AI任务使用独立锁、独立配额和独立健康状态，不得占用旧单股AI的轮次锁或完成记录；
- 雷达AI失败、未配置或达到配额时，只停止新模型调用，不回滚雷达规则结果，也不阻断下一轮确定性扫描。

## 14.1 盘中同时间比较

盘中累计成交额、成交量、换手、上涨广度和ETF流动性只能与历史相同交易阶段、相同已交易时长进行比较。上午早盘数据不得与完整交易日数据直接比较，也不得把午间静止期当作连续交易时间。

规则要求：

- 集合竞价、连续竞价、午间休市、收盘集合竞价和盘后交易分别标记；
- 配置每类指标的首个有效计算时点和最小历史样本量；
- 盘中放量使用过去若干交易日同一时间点累计值的中位数或稳健基准；
- 早盘、午后和尾盘可以使用不同季节性基线；
- 当日停牌、临时停牌、异常开盘和数据中断不进入正常基线；
- 行业、个股、ETF和基准使用同一 `as_of` 批次；
- 历史同时间样本不足时显示样本不足，不推算全天结果；
- 收盘后再生成独立的完整交易日指标，不用盘中估算值覆盖。

“放量、缩量、资金活跃和行业扩散”等盘中结论必须显示比较窗口和样本数，不能只给标签。

# 18. 历史回放与校准

## 18.1 核心要求

回放必须按历史时点推进，只能使用当时已经存在的数据。

必须处理：

- 公告原始时间；
- 新闻原始时间；
- 数据抓取延迟；
- 当时行业成分；
- 停牌；
- 涨跌停；
- 一字板；
- 规则版本；
- 数据缺失；
- 未来数据检查。

现有提醒系统的真实历史回放仍须单独达到至少20条真实样本、覆盖10个日期，并补齐5分钟、30分钟、收盘、次日、5日及当时行业和指数基准。该工作不能用雷达20个交易日影子运行替代，雷达回放也不能冒充现有提醒回放。

## 18.2 评估指标

### 行业

- 主线持续天数；
- 一日游误报；
- 分歧后回流识别；
- 退潮识别延迟。

### 龙头

- 预备升级率；
- 候选确认率；
- 已确认失效率；
- 假龙头率；
- 高位误报率；
- 3日和5日相对行业表现；
- 最大不利波动；
- 状态稳定性；
- 空榜比例。

### ETF

- 5、10、20日表现；
- 1、3、6个月表现；
- 相对基准；
- 最大回撤；
- 换手和成本敏感性。

## 18.3 最低正式门槛

雷达从影子转正式前至少满足：

- 连续20个A股交易日影子运行；
- 覆盖强势、震荡和退潮中的至少两种环境；
- 未来数据违规为0；
- 状态重复和同股多状态为0；
- 数据异常时不会生成伪结果；
- 关键规则拥有自动测试；
- 用户完成结果抽查。

达到功能验收不等于承诺选股收益。

20个交易日只是最短观察期，不代表到期自动转正式。影子开始前必须冻结并版本化本轮评估口径、状态阈值和人工抽查标准；运行期间不得为了让结果更好看而改变口径。若期间修改核心规则，相关观察周期应按新版本重新计算或分版本报告。

ETF两类周期分别验收：

- 趋势轮动观察：可以使用严格时间点历史回放和不少于20个交易日的影子运行进行第一阶段判断。
- 中期跟踪：1、3、6个月指标必须具有相应长度的严格时间点历史回放或真实观察期；未覆盖完整周期前只能保持影子状态，不得随趋势轮动功能一起自动正式开放。

正式转正报告必须分开披露数据完整性、状态稳定性、误报、空榜和相对基准表现，不得只展示正向样本或单一收益数字。

## 18.4 历史时点股票池与偏差控制

历史回放必须重建每个 `as_of` 时点真实可见的研究世界：

- 使用当时已经上市且尚未退市的证券名单；
- 保留后来退市、风险警示和被合并的证券，禁止幸存者偏差；
- 使用当时有效的板块交易规则、行业分类、指数成分和ETF成分；
- 使用公告和财务数据真实披露时间，不按报告期结束日提前使用；
- 使用当时生效的复权因子、股本、自由流通股本和公司行为；
- 规则版本、参数和人工映射必须按当时版本冻结；
- 缺少历史行业成分、分钟行情或公司行为时标记不可验证，不使用当前数据补写；
- 调参样本与最终保留验证样本分开，禁止在同一批结果上反复调参后仍称为独立验证。

回放结果必须同时报告纳入、排除、缺失和不可验证数量。只展示后来表现好的行业、ETF或股票属于无效验证。

## 18.5 独立龙头黄金样本

系统状态不能成为自己的正确答案。正式验证必须建立独立于规则输出的黄金样本集。

黄金样本要求：

- 覆盖强势、震荡、分歧、回流、退潮和风险环境；
- 同时包含真正行业核心、价格强但业务弱、单股独涨、跟风、伪关联、不可交易和应当空榜的案例；
- 在每个历史 `as_of` 时点，只向标注者提供当时已经存在的证据；
- 分别标注市场领先性、行业贡献度、业务暴露、可交易性、风险和不确定项；
- 事后3日、5日及更长表现只作为独立结果指标，不反向改写当时事实标签；
- 保存标注人、标注时间、证据来源、分歧和复核记录；
- 规则开发样本、参数校准样本和最终保留样本分开；
- 系统误报、漏报和人工无法判断都要保留，不能只留成功样本。

黄金样本至少用于检查：候选覆盖率、误报率、漏报率、状态提前或滞后、行业归属错误、业务暴露误判、可交易性误判和空榜正确性。样本不足时只能报告研究结果，不得宣称龙头识别已经有效。

行业收益的基础公式固定为：

```text
equalReturn = mean(constituentReturn)
capWeight_i = marketCap_i / sum(marketCap)
capReturn = sum(capWeight_i × constituentReturn_i)
contribution_i = capWeight_i × constituentReturn_i
exTopReturn = sum(otherContribution) / sum(otherCapWeight)
```

第一贡献股按本轮 `contribution_i` 最大值确定，不按股票涨幅或市值单独猜测。若剔除后没有有效成分，`exTopReturn` 为缺失。不同供应商的行业涨跌、资金、公司家数与本系统按版本化成分计算的等权/市值/广度必须分别保存、分别展示，不相减、不共用分母。

#### 分数、完整度和状态门禁

- 影子分数最少需要相对强度、成交额放量、上涨广度、持续性四个核心维度全部有效，即 `participatingWeight >= 75`；任一核心维度缺失时只返回特征和缺失原因，不返回可比较总分；
- 涨停扩散、催化和ETF属于独立确认维度。未验收时保持缺失，不会把75分核心权重重标成100分；
- 同一行业只有在规则版本、分类版本、市场批次和全部来源身份一致时才可跨时点比较；不同版本分数不得直接连成同一历史曲线；
- “观察、启动、确认、加速、分歧、回流、退潮、失效”的数字阈值本阶段不批准。后续必须基于冻结合同至少20个交易日的数据，覆盖至少两种市场环境，分离校准样本与保留样本后再提交；
- 阈值方案必须分别给出进入、保持和退出值、连续次数、最短保持时间、冷却期以及数据失败行为。不能只给一个分数切线，也不能在影子运行中为改善结果临时调参；
- 行业状态不得由新闻、ETF、AI或单只第一贡献股单独触发。AI仍只解释冻结的确定性结果，不能修改行业、分数、排名或状态。

#### 数据量、存储与回退审核

- 按每3分钟一轮、每个交易日约80轮、最多100个行业估算，行业聚合约8,000行/日，20个交易日约160,000行；若每行含合同、特征和证据摘要约1至2KB，SQLite原始增量约160至320MB，另计索引和备份；实施迁移前必须用真实序列化结果重新实测；
- SQLite只保存市场和行业聚合、规则版本、状态及必要冻结证据，不保存5,529只证券的逐轮原始行情；需要严格原始快照回放时仍须在阶段9前单独提交Parquet等方案；
- 本阶段没有数据库和运行时变更，不需要生产数据回退。后续迁移必须先创建一致性备份，在复制库完成迁移和恢复演练，再单独申请生产迁移；功能回退先关闭新增市场/行业影子开关，不删除已产生历史。

| `classificationStartDate` | 指引规定的当期分类工作起始日 | 用于解释当期新上市公司是否应被覆盖，不等于结果生效日 |
| `sourceSymbol` / `sourceName` | PDF原始代码和简称 | 原样保留；简称只校验，不作为连接键 |
| `securityIdentity` / `identityStatus` | 证券生命周期解析后的稳定身份；状态为 `exact`、`verified_alias` 或 `unresolved` | 只允许代码精确命中或有公布/生效证据的别名；禁止名称和模糊匹配 |
| `categoryCode` / `categoryName` | 门类代码和名称 | 作为主行业父级，必须同时非空且代码名称一一对应 |
| `divisionCode` / `divisionName` | 大类代码和名称 | 作为雷达统一主行业，必须同时非空且代码名称一一对应 |
| `manufacturingSubclassCode` / `manufacturingSubclassName` | 只适用于制造业的次类 | 非制造业必须为空；制造业缺失或代码名称冲突时该行拒绝正式接纳 |
| `middleClassCode` / `middleClassName` | 当前公开结果没有该层级 | 固定为空并标记 `source_not_published`，不得推断 |
| `recordStatus` / `issueCodes` | `accepted`、`unconfirmed`、`conflict`、`source_failed`及明确原因 | 未确认和冲突记录不参与正式成分、分数或状态 |

原始发布粒度主键为 `classificationSystem + releasePeriod + sourceSymbol`；解析身份后的唯一约束为 `classificationSystem + releasePeriod + securityIdentity`。同一版本内每个稳定证券身份必须恰好一个大类；重复、必填字段缺失、代码名称一对多、哈希变化、未知层级、未验证别名和未来版本均使相关记录或整批降级。

历史补录与实时版本必须分开标记。2025年上半年和下半年文件是在本轮才首次取得，能够证明当前官方档案中存在这些版本并支持差异研究，但不能仅用当前下载哈希证明文件在原公布日之后从未改变；因此两份历史样本当前只允许作为 `retrospective_unverified` 研究证据，不得在黄金样本回测中自动回填为当时已知事实。适配器上线后新发现的版本从首次成功观察时点开始可进入向前运行。
