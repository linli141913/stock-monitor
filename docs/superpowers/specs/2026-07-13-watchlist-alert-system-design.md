# 关注股票强提醒系统设计

日期：2026-07-13

状态：总体方案与换手率口径已确认；按小功能分批实施

适用项目：`/Volumes/HermesSSD/AntigravityData/量化监测-股票`

执行说明：本文是强提醒系统的唯一总体设计。换手率三档仅属于第 6.2.1 节的量价风险子功能，不能替代官方重大事件、站内提醒、邮件送达、监测健康和数据纠错等完整闭环。

本文同时包含总体设计、实施顺序和验收标准，作为本次升级的唯一执行文档。

## 1. 目标

在不更换现有 Next.js、FastAPI、SQLite、APScheduler 技术框架的前提下，为后台监测列表中的最多 10 只证券建立完整提醒闭环。监测范围包括 A 股公司、港股公司和沪深交易所上市的国内 ETF：

1. 发现官方重大正面、重大负面和重要中性事件。
2. 在没有新闻或公告时，通过量价、资金和板块数据识别风险升高或正向异动。
3. 把提醒持久保存到站内提醒中心，并将高优先级提醒发送到用户配置的邮箱。
4. 保存原始证据、规则、数据时间和送达状态，能够解释为什么提醒、何时提醒、是否成功送达。
5. 用时间截断验证规则，避免把事后解释包装成事前预警。

系统只分析事件、局势、影响、风险和待验证证据，不提供买卖、加减仓、目标价或确定性价格预测。

## 2. 非目标

第一阶段不做以下内容：

- 对全市场每只股票逐一生成提醒或逐份公告做深度解析；行业洞察的全市场资讯雷达不属于逐股提醒扫描。
- 自动交易或券商下单。
- 微信、短信、Telegram 等新通知渠道。
- Web Push 离线推送和原生 App。
- 毫秒级或秒级高频预警。
- 以 AI 评分直接触发提醒。
- 在缺少历史数据时伪造连续资金流或回填假数据。

## 3. 当前问题与设计约束

### 3.1 已确认的问题

- 新闻采集、AI 分析和页面展示已经存在，但没有“分类、生成提醒、送达、确认、重试”的完整链路。
- 巨潮公告目前使用股票代码全文搜索，并可能把其他公司的搜索结果强行关联到目标股票，存在严重误匹配风险。
- 个股相关新闻标签会优先显示全局新闻，导致无关资讯淹没个股信息。
- 提醒卡和提醒页面目前是禁用占位，没有真实保存或发送能力。
- AI 分析侧重事后归因，前端的多头共振评分和未来走势推演容易被误解为投资建议。
- 后台监测列表与用户主观“关注过这只股票”不是同一状态，页面没有明确告诉用户后台订阅是否真正生效。

### 3.2 必须保留的约束

- 前端本地端口保持 `4000`，后端保持 `8001`。
- 继续使用服务端代理保护后端 Token，前端不保存私密凭证。
- 数据源失败时显示失败或不可用，不用旧数据或 0 冒充新数据。
- 原始数据时间、系统抓取时间、规则触发时间和页面渲染时间必须分开。
- 数据库迁移前必须备份，迁移失败可以回退。
- 新提醒模块调用内部业务函数，不新增依赖于固定本机 HTTP 端口的后端自调用。

## 4. 总体架构

提醒链路固定为：

```text
公开数据源
  -> 采集与标准化
  -> 股票精确关联
  -> 事件分类 / 指标计算
  -> 规则引擎
  -> 提醒事件入库
  -> 站内提醒 / 邮件发送
  -> 送达、已读、失败与重试记录
```

AI 只在规则已经触发之后生成通俗解释、影响情景和待验证证据。AI 不决定是否触发，不得提高官方证据等级，也不得把未验证消息改写成事实。

## 5. 提醒模型

每条提醒包含三个互相独立的维度。

### 5.1 方向

- `positive`：重大正面事件或正向市场异动。
- `negative`：重大负面事件或风险升高。
- `neutral`：重要但方向不明确。
- `uncertain`：证据不足，暂无法判断。

方向不等于未来股价预测。“正面”只表示事件本身可能改善预期，“负面”只表示事件本身可能增加风险。

### 5.2 优先级

- `P1 紧急`：立即站内提醒并发送邮件。
- `P2 重要`：立即站内提醒；是否即时发邮件由用户偏好决定，默认发送。
- `P3 观察`：站内记录并进入摘要，不进行强打扰。

重大正面和重大负面都可以是 P1，优先级只表示需要关注的紧迫程度，不表示涨跌方向。

### 5.3 证据等级

- `S`：交易所、公司、监管部门等官方原文。
- `A`：两个及以上相互独立的可靠来源，或来源可靠且有可核验原始数据。
- `B`：单一可信媒体或机构观点。
- `C`：来源可追溯但尚未独立印证的线索，仅允许进入观察层，不允许触发 P1。

`C` 是产品展示下限，不再设置面向用户的 D/E 等级。来源不明、原文链接缺失且无法追溯、内容无法与目标股票或行业准确关联的资讯，内部标记为“未评级/拒绝”，不进入行业洞察展示、AI 分析和提醒；原始抓取记录可保留用于去重与审计。

## 6. 第一阶段事件规则

### 6.1 官方公司事件

所有与监测股票精确匹配的官方公告至少生成一条 P2 或 P3 站内提醒。以下类型默认提升优先级：

负面或风险类：

- 股票交易异常波动和特别风险提示。
- 业绩预亏、业绩大幅下降、重大减值。
- 立案调查、监管处罚、重大问询。
- 债务逾期、信用下调、重大诉讼。
- 股东或管理层重大减持。
- 停产、项目终止、合同终止、重大客户流失。
- 可能影响持续经营的重要人员离任。

正面或催化类：

- 业绩预增或明显超出已有公开口径。
- 重大合同、中标、订单或长期供货协议。
- 产品、药械、技术或项目获得正式批准。
- 重要回购、增持、融资支持。
- 重大项目投产、产能落地或合作确认。

中性但重要：

- 分红、定增、股权激励、重大资产交易。
- 董事会换届、公司治理和组织变更。

规则分类只生成初步方向和优先级；提醒正文必须保留“可能影响”和“待验证”，不得把事件直接解释为确定性涨跌。

### 6.2 量价风险与正向异动

初始候选指标：

- 10 日涨幅不低于 30%。
- 20 日涨幅不低于 50%。
- 单日振幅不低于 12%。
- 换手率相对该股票自身近期常态明显升高。
- 量比不低于 2。
- 连续资金流出、价格与资金背离；只有真实历史资金数据完整时才启用。
- 放量跌破 MA5、MA10 或 MA20。
- 个股与所属板块出现明显背离。
- 涨跌停或单日涨跌幅达到极端区间。

普通情况下，至少两个相互独立的指标同时触发，才生成 P2 风险或正向异动提醒。单一指标只生成 P3。涨跌停、官方特别风险提示等强事件可以直接生成 P1。

“正向异动”不能写成“利好确认”；只有官方或已交叉验证的公司事件才允许标记“重大正面”。

#### 6.2.1 换手率三档提示

换手率本身已经按成交股数与流通股数的比例计算，能够反映股票流通盘大小。系统不再用“所有股票超过同一个百分比”的方式判断，而是把当前换手率与该股票自己的历史常态比较。大盘股平时换手率很低，即使当日只有 1%，也可能已经属于明显活跃或需要警惕；小盘股即使达到较高百分比，也可能仍接近自身日常水平。

盘中累计换手率不能直接与历史收盘换手率比较，否则上午天然偏低、下午天然偏高。核心指标必须按相同交易时点比较：

```text
盘中换手活跃倍数 = 当前累计换手率 / 近 20 个有效交易日同一时点累计换手率中位数
收盘换手活跃倍数 = 当日换手率 / 近 20 个有效交易日收盘换手率中位数
```

盘中“同一时点”按最接近的 5 分钟快照对齐，午间休市期间沿用上午收盘时点，不把休市时间当成交易进度。使用中位数而不是平均数，避免某一天的极端换手率把基线明显抬高。流通市值、成交额和成交额相对近 20 日同一时点中位数的倍数作为解释和辅助信号，不替代个股自身基线。

三档规则：

- `正常`：换手活跃倍数低于 1.5，且没有其他明显量价异常。
- `活跃`：换手活跃倍数达到 1.5、但低于 2.5；表示交易明显升温，不代表利好、利空或价格方向。
- `警惕`：换手活跃倍数不低于 2.5，并同时满足至少一项辅助条件：成交额达到近 20 日中位数的 2 倍、量比不低于 2、当日振幅不低于 8%、当日涨跌幅绝对值不低于 5%、出现可验证的明显资金流出。

特殊处理：

- 有效历史换手率不足 20 个交易日时显示“样本不足”，不得显示“正常”。
- 没有经过验证的历史盘中换手率或同一时点快照时，不得用历史收盘换手率冒充盘中基线；先积累真实快照或接入可追溯的历史分钟数据。
- 数据源缺失、过期或口径未确认时显示“暂无判断”，不得根据 0 值判断。
- 第一阶段只对已验证口径的 A 股启用；港股在历史换手率和流通盘口径验证通过后再开放。
- `活跃`只生成 P3 观察记录；`警惕`由于已经包含换手异常和至少一个辅助信号，可以生成 P2 风险提醒。
- 换手率提示只描述筹码交换活跃程度，不得写成主力吸筹、出货或未来涨跌结论。

页面必须同时显示颜色和文字，不能只依赖颜色传达状态：

- `正常`：蓝灰色状态标签，文字“正常”。
- `活跃`：橙色状态标签，文字“活跃”。
- `警惕`：红色状态标签，文字“警惕”。
- `样本不足`或`暂无判断`：灰色状态标签和对应文字。

换手率区域示例：

```text
换手率（动） 1.00%  [警惕]
今日为近 20 日换手率中位数的 5.6 倍，并伴随成交额异常放大。
```

示例中的数字只说明展示格式，实际倍数必须由真实历史数据计算。

### 6.3 板块与海外联动

第二阶段启用：

- 所属板块跌幅不低于 3%。
- 板块上涨家数占比低于 20%。
- 板块龙头跌幅不低于 8%或触及跌停。
- 板块资金流进入流出前列。
- 与公司业务有明确映射的海外指数或核心公司出现极端波动。

海外信号必须先经过业务映射，不允许仅因“都是科技股”就强行关联。

### 6.4 监测系统健康提醒

以下情况生成系统提醒，防止系统静默失效：

- 同一数据源连续两个周期抓取失败。
- 监测股票超过预期时间没有更新。
- 浏览器监测列表和后端监测列表不同步。
- 邮件连续重试后仍发送失败。
- 股票代码或公司名称无法精确映射。

### 6.5 行业洞察与首页政策动态

行业洞察和首页“相关政策 / 上下游动态”统一使用提醒中心的三个独立维度，不把来源、影响方向和重要程度混成一个标签：

- 来源等级：`S / A / B / C`。
- 影响方向：`正面 / 负面 / 中性 / 待核验`。
- 重要程度：`P1 重大 / P2 重要 / P3 一般`。

证据不足时必须显示“待核验”，不得硬判利好或利空。AI 单独判断不得把事件提升为 P1；P1 必须有官方原文或多源独立证据。收起状态展示当前最高优先级事件的三个标签和标题，展开后每条资讯分别展示标签、来源、发布时间和原文链接。

来源核验状态不再统一写死为“未独立交叉验证”：官方原文显示“来源已核验”，两个及以上独立来源显示“多源印证”，单一可靠来源显示“单一来源”，C 级显示“线索级，待核实”。来源真实性与影响判断分开表达。

行业洞察顶部分类精简为“全部 / 公司公告 / 行业政策 / 产业动态 / 海外与管制”。“国内 / 国外”作为资讯标签，不再单独占据一级分类；删除未真正实现过滤的“只看半导体”和未验证公司地位的“龙头公司”分类。

行业洞察使用独立的全市场资讯池，覆盖与 A 股、港股和沪深交易所国内 ETF 有关的公司重大公告、行业政策、产业动态、海外市场与管制信息，不受当前监测列表限制；同时保留证券市场相关性过滤，普通社会新闻和无法关联资本市场或产业的国际新闻不得进入。提醒中心继续只服务监测列表，首页“相关政策 / 上下游动态”继续只展示当前股票的准确关联信息。

提醒中心、行业洞察、首页“相关政策 / 上下游动态”统一执行严格的当天数据规则：来源提供精确发布时间时，只接受上海时区当天的资讯；来源只提供公告日期或午夜日期占位时，以系统首次发现日作为实际展示日，当天发现当天展示，第二天不再展示。官方归档日期允许与首次发现日相同或晚一天；若官方日期早于首次发现日，说明属于迟到旧闻，直接拒绝。昨天及更早的资讯即使今天才被系统首次抓到，也不得生成提醒、进入行业洞察或从旧缓存回填首页；来源发布时间缺失时同样不得展示。系统发现时间只用于校验日期精度来源和审计，不能把旧资讯变成新资讯。当天没有真实、可追溯、准确关联的资讯时显示空状态，不得用历史数据补位。

提醒中心默认展示“今日提醒”，另设“历史记录”用于回看过去已经实际触发并持久保存的提醒。股票后续移出监测列表时，已经生成的历史提醒仍保留可查；历史记录不把迟到旧闻重新生成提醒，也不影响今日未读数量，不修改数据库结构。

### 6.6 A 股、港股和国内 ETF 的动态关联

- A 股公司、港股公司和国内 ETF 是三类独立资产；国内 ETF 不得按港股代码或港股交易状态处理。
- 新加入监测列表的公司股票按证券代码和真实公司名称抓取定向资讯，再解析其真实行业标签；新加入的国内 ETF 按基金名称、指数或主题建立搜索词。
- 不再把无法识别行业的证券默认归入半导体，也不得因为系统最初用于半导体监测而保留无关半导体资讯。
- A 股公告必须通过证券代码精确匹配官方结果。港股公告汇总和国内 ETF 基金公告按证券代码精确查询；若原文不是交易所或监管官方域名，只能标记为 B 级“公告汇总”，不得伪装成 S 级官方原文。
- 定向媒体资讯必须保留真实来源、原始发布时间和可追溯链接。分类、三个标签、P1/P2 提醒、事件 AI 去重以及四个定时复盘时点继续使用同一套提醒方案。

## 7. 精确关联与数据正确性

公告进入提醒系统前必须满足以下条件之一：

1. 官方返回的 `secCode` 与目标股票代码完全一致。
2. 官方返回的证券内部 ID 与预先保存的目标证券 ID 一致。

不得再把全文搜索结果中的所有记录统一写成目标股票代码。股票名称模糊匹配只能用于发现候选，不能用于生成高优先级提醒。

每条原始资讯至少保存：

- 原始事件 ID。
- 证券代码和证券名称。
- 来源和原文链接。
- 标题和原文发布时间。
- 系统首次抓取时间。
- 匹配方法和匹配置信度。
- 原始分类和处理状态。

## 8. 数据库设计

实施时先备份现有 SQLite，再通过可回退迁移新增以下表。

### 8.1 `monitoring_preferences`

保存每只股票是否启用提醒、邮箱渠道、P2 是否即时邮件、各类事件开关和更新时间。

### 8.2 `signal_snapshots`

保存计算风险规则所需的真实指标快照、原始数据时间、抓取时间、数据完整性和来源状态。

### 8.3 `alert_events`

保存提醒方向、优先级、证据等级、触发规则、关联股票、摘要、原文链接、首次触发时间、当前状态和去重键。

### 8.4 `alert_deliveries`

保存站内和邮件渠道的发送时间、成功或失败状态、失败原因、重试次数和最终结果。

### 8.5 `alert_email_settings`

只保存一个全局强提醒收件邮箱。用户可在页面上随时修改并覆盖上一个地址；修改后的新提醒发往新地址，历史提醒和既有送达记录不改写。SMTP 发件凭证仍只保存在后端环境变量，不进入浏览器或数据库。

## 9. 去重、升级与重试

- 去重键由证券代码、原始事件 ID、事件类型和来源组成。
- 同一官方事件默认只发送一次。
- 如果优先级从 P2 升到 P1，或出现新的 S 级证据，允许发送升级提醒。
- 量价信号设置冷却期；指标仍处于同一风险区间时不重复轰炸。
- 指标退出风险区间后再次进入，可以生成新提醒。
- 邮件失败后按 1 分钟、5 分钟、15 分钟最多重试三次。
- 最终失败必须保留站内提醒，并显示“邮件未送达”。

## 10. 调度频率

在最多 10 只监测股票的范围内采用以下默认频率：

- 官方公告：07:00 至 23:00 每 5 分钟，其他时间每 30 分钟。
- 交易时段个股量价风险：每 60 秒。
- 交易时段板块风险：每 3 分钟。
- 普通媒体新闻：每 10 分钟。
- AI 解释：P1/P2 新事件出现后按需运行，同时保留监测列表每天 10:30、11:30、15:00、22:00 四次定时复盘；定时复盘不得通过同步 HTTP 请求后端自身，必须在后台线程执行，不能阻塞 FastAPI 主事件循环。
- AI 定时复盘：同一股票、同一时间点只允许执行一次；上一轮未结束时跳过新一轮，不排队，不自动重试。页面重复刷新在 20 分钟内复用最近一次成功结果；同一 P1/P2 事件只允许额外分析一次。
- AI 历史复盘：从历史日历选择记录后进入历史查看模式，隐藏“今日盘中追踪时间轴”，顶部显示由后端交易日历返回的真实分析周期起止时间；关闭历史周期标签后恢复当前周期记录，不得通过重新调用 AI 才能退出历史。
- 监测健康审计：每次采集后更新，每天收盘后生成一次完整检查结果。

## 11. 后端模块边界

不继续把新逻辑堆入 `backend/main.py`。新增模块职责如下：

- `event_classifier.py`：标准化事件、分类方向、计算证据等级。
- `alert_engine.py`：执行固定事件和量价规则，生成提醒事件。
- `notification_service.py`：站内写入、邮件发送、重试和送达记录。
- `alert_repository.py`：提醒、偏好、快照和送达记录的数据库访问。

`main.py` 只注册路由和调度任务。采集、规则和通知模块可以独立测试。

## 14. 错误处理

- 采集失败：保存失败状态，不生成伪事件。
- 证券无法精确匹配：进入待核验队列，不给用户发送 P1/P2。
- 指标缺失：只计算可用指标，并在提醒中显示数据完整性；不得用 0 补缺失值。
- AI 调用失败：保留规则提醒和原始证据，显示“AI 解释暂不可用”。
- 邮件失败：站内提醒仍然有效，并按重试策略处理。
- 数据过期：明确标记过期，不继续把它显示为当前状态。
